# Тест-план - F-Bank

## 1. Объект тестирования

Веб-приложение **F-Bank** - учебный банковский онлайн-сервис для перевода денежных средств со счёта на карту. Приложение представляет собой одностраничный SPA на React, развёрнутый локально по адресу `http://localhost:8000/`.

Начальный баланс рублёвого счёта и сумма "резерва" задаются параметрами URL:
`http://localhost:8000/?balance=<целое число>&reserved=<целое число>`

Если параметры не заданы, баланс рублёвого счёта = 0.
Доллары (100 usd, резерв 0) и евро (300 eur, резерв 26) - захардкожены.

## 2. Функциональность под тестированием

| № | Функция | Описание |
|---|---------|----------|
| F1 | Отображение счетов | Три карточки: рубли, доллары, евро. Каждая показывает баланс и резерв |
| F2 | Выбор счёта | Клик по карточке счёта открывает форму перевода |
| F3 | Ввод номера карты | Поле "Номер карты:" с маской `0000 0000 0000 0000` |
| F4 | Ввод суммы перевода | Поле "Сумма перевода:" с дефолтным значением 1000 |
| F5 | Расчёт комиссии | Комиссия = `floor(сумма / 100) * 10` |
| F6 | Проверка достаточности средств | При недостатке - ошибка "Недостаточно средств на счете", кнопка "Перевести" скрыта |
| F7 | Подтверждение перевода | Кнопка "Перевести" → alert "Перевод X rub на карту Y принят банком!" |

## 3. Стратегия тестирования

- **Чёрный ящик / функциональное тестирование**: проверка позитивных и негативных сценариев со стороны UI.
- **Граничные значения**: длина номера карты, нулевая/отрицательная сумма перевода, точная сумма к остатку.
- **Кросс-валютная проверка**: поведение проверки достаточности средств при выборе разных валют.

## 4. Окружение

- ОС: macOS 14+ / Ubuntu 22.04 (CI)
- Браузер: Google Chrome (последний)
- Сервер приложения: `python3 -m http.server 8000` из каталога `dist/`
- Тестовый URL для большинства тест-кейсов: `http://localhost:8000/?balance=10000&reserved=0`

## 5. Критерии входа / выхода

**Вход**: приложение собрано, статика отдается с `localhost:8000`, страница открывается без ошибок в консоли.

**Выход**: выполнены все пять тест-кейсов, найдено не менее двух различных дефектов, по каждому дефекту составлен баг-репорт, на каждый дефект написан автотест.

## 6. Риски

- Состояние счетов (USD/EUR) задано в коде - недоступно для конфигурации через UI, что ограничивает количество позитивных сценариев.
- Параметры URL `balance`/`reserved` не валидируются - возможно появление дополнительных дефектов при аномальных значениях (вне scope настоящего плана).

## 7. Распределение тест-кейсов

Всего 5 тест-кейсов (см. [test-cases.md](test-cases.md)):

| ID | Тип | Ожидаемый результат |
|-----|-----|---------------------|
| TC-01 | Позитивный | Перевод проходит |
| TC-02 | Негативный | Появляется ошибка "Недостаточно средств" |
| TC-03 | Негативный (дефект) | Поле номера карты не должно принимать > 16 цифр |
| TC-04 | Негативный (дефект) | Отрицательная сумма не должна приниматься к переводу |
| TC-05 | Негативный (дефект) | Проверка достаточности средств должна учитывать выбранную валюту |
